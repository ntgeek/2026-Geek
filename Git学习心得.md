## 学习内容  
### Git的工作区域和流程  
1. 工作区：本地编辑文件的目录
​
2. 暂存区：临时存放待提交的修改（ git add ）
​
3. 本地仓库：保存完整版本历史（ git commit ）
​
4. 远程仓库：进行托管的仓库（ git push/pull ）  
### 本地仓库操作
 
1. 初始化仓库： git init ，在当前目录创建一个新的 git 仓库
​
2. 添加到暂存区： git add 文件名  或  git add . （这个可以添加所有修改）
​
3. 提交到本地仓库： git commit -m "提交说明" ，必须写清晰的备注，方便回溯版本  
### 远程仓库关联与同步
 
1. 克隆远程仓库： git clone 仓库地址 ，将想要的在GitHub中的仓库下载到本地
​
1. 关联远程仓库： git remote add origin 仓库地址 ，建立本地与远程的连接
​
1. 推送本地提交： git push origin 分支名 ，将本地版本同步到云端
​
1. 拉取远程更新： git pull origin 分支名 ，将云端最新版本同步到本地
 ***
### 分支管理与冲突解决
 
- 创建分支： git branch 分支名 ，用于开发新功能/修复bug，不影响主分支
​
- 切换分支： git checkout 分支名  或  git switch 分支名 
​
- 合并分支： git merge 分支名 ，将子分支内容合并到当前分支
​
- 冲突解决：当多人修改同一文件时，Git 会标记冲突，手动编辑后重新  add/commit  
### 感想   
一开始学命令的时候，是非常懵的， add   commit   push   pull  这些指令记了又忘，有时输错命令就会很不知所措，不知道怎么回退，怎么解决。但通过这些天对于Git和GitHub的接触，才发现Git 的核心逻辑特别简单——就是记录每一次文件的修改，想回到哪个版本、想对比哪次改动，都能随时查，不用靠记忆存文件版本了。
 
这几天又新了解到了Git在小组协作写代码工作的作用，觉得Git功能很强大。我了解到多人一起改同一个项目，可能会出现“我改了这个文件，他也改了，最后大家的版本都覆盖了”的问题，大大影响了进度。而用Git之后，每个人在自己的分支上改代码，提交后再合并到主分支，就算改了同一行也能清晰看到冲突提示，解决冲突的过程也相对轻松。
 
当然，Git也不是完全没门槛，特别容易上手。比如远程仓库的连接、PR的提交流程，一开始操作的时候总容易出错，这都需要耐心和时间一步步去理解和掌握。
 
总的来说，Git 不是那种“学完就忘”的工具，越实操越能提高掌握度，后续我也想再深入学学分支管理的高级用法，能让自己的代码与他人的协作更高效。
